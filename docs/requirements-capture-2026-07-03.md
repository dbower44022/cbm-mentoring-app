## Identity & Access (5)

- **REQ-004 — Act-as-User CRM Access** (must): The application performs every CRM read and write as the authenticated user's own CRM account, never through a shared elevated credential for user-initiated operations. CRM role and record-level permissions therefore govern what each user can see and change.
- **REQ-005 — Server-Side Session Management** (must): Login sessions are managed server-side: the browser holds only an opaque session reference, and expiry, idle timeouts, and revocation are enforced at the server. When a session expires, every open window offers re-authentication in place, unsaved work survives, and one re-login restores all windows. Logout is explicit and ends access across all windows.
- **REQ-006 — Per-Data-Source Access Grants** (must): Access to information is granted per data source: a user can open only panels and views whose underlying data source they have been granted, and that grant is the approval boundary for what they can see. Data sources may declare a user-based row filter that the server injects and the user cannot bypass.
- **REQ-007 — Token-Based One-Off Actions** (should): The platform supports signed, expiring action links that let a recipient perform one specific action without logging in. A token is bound to one identity and one named action, validated and use-accounted server-side, and every mint, redemption, and revocation is recorded.
- **REQ-008 — Pluggable Authentication Extensibility** (could): Credential verification is a pluggable step that returns a verified identity, decoupled from session establishment, so single sign-on and multi-factor checks can be added without re-architecture. The login surface reserves space for a single sign-on affordance and for factor prompts when those ship.

## Application Shell & Navigation (7)

- **REQ-009 — Standard Application Header** (must): Every window carries one thin standard header: identity and navigation on the left; notification bell, Help, and the user menu on the right. The user menu is the single home for per-user preferences, including navigation style, startup choice, themes, view and pin management, and logout. A quick-open palette (Ctrl+K) offers type-ahead access to every panel and view the user can reach.
- **REQ-010 — Panel Navigation and Pinning** (must): The application is composed of permissioned panels, grids being the most common type. Each user chooses how navigation renders — tabs across the top, items down the side, or a group tree for large sets — switchable anytime without losing pins. The system defines default tabs and menus, and users pin their own views as personal navigation entries.
- **REQ-011 — Home Panel and Admin Messaging** (must): A provided Home panel combines messages from the system administrator with dashlets the user chooses. Messages carry per-user read state with unread counts, optional acknowledgment tracking visible to the admin, and an urgent priority that banners across every panel until read. Startup is a per-user preference: open to Home or to the last panel, with Home as the system default.
- **REQ-012 — Multi-Window Record Viewing** (must): Records open wherever possible in a docked preview pane or a non-modal pop-out rather than navigating away, so the user keeps working the grid. Pop-outs are real browser windows pinned to their record, several may be open at once, and the docked preview always follows the currently selected row. Previews are read-optimized, never edit forms.
- **REQ-013 — Cross-Window Freshness and Edit Collision** (must): For the same user, a save in any window immediately updates every open window showing that record or a grid containing it. Closing a window with unsaved edits triggers a dirty-window guard, and invoking Edit on a record already being edited offers switching to the existing editor instead of opening a second one. Saves verify the record is unchanged since load and walk the user through conflicts rather than overwriting.
- **REQ-014 — Background Task Notifications** (must): Any action that can run longer than about ten seconds executes as a background task — never a spinner that traps the user. The panel stays free, the status bar shows progress while the user remains on it, and a header notification bell collects completions and failures per user, marked read on view and expiring over time.
- **REQ-015 — Broken Pin and Startup Fallback** (must): A pinned view or panel that becomes unavailable is never silently removed: the entry stays, subtly marked, and clicking it explains what happened — what was deleted or revoked, by whom, when — with choices to remove the pin or pick a different view. Deep links and the open-to-last-panel startup follow the same rule, falling back to Home with the explanation instead of a blank screen.

## Grid Platform (16)

- **REQ-016 — Universal Grid Anatomy** (must): Every grid, without exception, is three stacked regions: an action bar above, the data table, and a status bar below. Grids use infinite scrolling with cache optimization — there is no page size anywhere. On wide screens columns expand automatically so the grid shows no white space unless there is too little data to fill it.
- **REQ-017 — Grid View System** (must): Each grid opens on the view last used by that user and offers a view selector holding system-defined and user-defined views. System views are read-only: a user modifies one in the view editor and either saves it as their own view or applies it temporarily, with the temporary state marked in the selector until another view is chosen. Users manage their own views; admins may promote a user view to a system view.
- **REQ-018 — View Definition Scope** (must): A view defines, in order: its data source; which fields display with column order, width, and format; grouping, including whether the grid acts as a tree and whether groups start collapsed; and a row theme covering row height, colors, and font. Header-driven sort changes and ad-hoc column filters mark the view as temporarily modified and can be saved into a user view.
- **REQ-019 — Admin-Authored Data Sources** (must): System administrators author the data sources all users consume, using both a visual query builder and raw SQL, with the power of joins, filters, and calculated fields. The fields a data source exposes determine what its views can display.
- **REQ-020 — Grid Search Box** (must): A search box sits in the middle of every grid's action bar. It searches on key press once three or more characters are typed, filters on top of the view's own filters without ever replacing them, scopes to the displayed columns, and remembers the last five searches per grid.
- **REQ-021 — Grid Actions Model** (must): Every data set ships create, read, update, and delete actions plus any functions and workprocesses defined for it. The action bar's right side holds the two most common actions and an Other Actions dropdown of everything; right-click shows the same list. Actions act on the selected rows and are never hidden or disabled — an invalid invocation explains in detail exactly why it cannot run.
- **REQ-022 — Destructive Action Confirmation** (must): Actions are classified at definition as safe, modifying, or destructive. Destructive actions always confirm, naming the action, the exact count, the first affected records, and any selected rows hidden by the current filter. Because the system never physically deletes records, confirmation wording is honest about recoverability — no false cannot-be-undone claims and no type-to-confirm theatrics.
- **REQ-023 — Grid Selection Semantics** (must): Single click selects a row and deselects others; shift and control click multi-select and reveal a checkbox column; double-click opens the record; select-all selects the entire filtered result set, not just visible rows. Changing the search or filter keeps the selection, with the status bar noting how many selected rows are outside the current filter, and action confirmations spell out that hidden selected rows are included.
- **REQ-024 — Grid Keyboard Standard** (must): One keyboard model covers every grid: arrow keys move row focus with a visible indicator, space toggles selection, shift-arrows extend, control-A selects the entire filtered set, enter opens the focused record, and the menu key opens the actions menu. Focus starts in the search box, slash jumps back to it, column-header focus plus enter sorts, and arrow-down at the bottom edge keeps loading.
- **REQ-025 — Multi-Column Sorting** (must): Header click sets the sole sort and repeat clicks toggle direction; shift-click adds secondary and tertiary sorts, with a third click removing that column. Sorted headers show a direction arrow and a numbered badge for position. Header sorting is a temporary view modification, flagged in the selector and saveable into a user view.
- **REQ-026 — Server-Side Aggregates and Status Bar** (must): Row counts and aggregates are computed server-side over the entire filtered result set, with aggregates shown on group rows and in a grid footer row for the whole set. The status bar shows the row count and selected count on the right, action progress messages in the middle, and a progress bar with an estimate whenever an action runs longer than a few seconds.
- **REQ-027 — Grid Export and Print** (must): Export and print are standard actions on every data set. Both operate on the selection if one exists, otherwise the entire filtered set, rendered as the current view shows it — columns, order, formats, sort, filters, and active search. Export produces CSV and Excel with formatted values by default and a raw-values option; long runs report progress in the status bar.
- **REQ-028 — Grid Deep Links** (should): A grid's URL identifies the grid and its active view, so a link reopens that panel with that view. Links are references, not grants: data-source permission is still required, a link to another user's private view falls back to the recipient's last-used view with an explanatory notice, and system-view links work for everyone with access.
- **REQ-029 — Per-Column Ad-Hoc Filters** (should): A view setting controls whether its grid offers ad-hoc column-header filters: distinct-value checkboxes, ranges, or contains conditions, indicated by a filled funnel. Ad-hoc filters combine with the view's filters and the search box, mark the view as temporarily modified, and can be saved into a user view. Distinct-value lists come from the server-side filtered set.
- **REQ-030 — Grid Empty and Error States** (must): Grids distinguish four states, each explaining what happened, why, and what to do next: an empty view states its criteria; a search filtered to zero says how many rows are hidden and offers clearing it; a data-source error speaks plainly with a retry control and detail available on demand; a permission refusal names the missing permission and who grants it.
- **REQ-031 — Grid State Restoration** (must): Returning to a grid from a record restores it exactly — view with temporary modifications, search text, scroll position, selection, and focused row — while the data refreshes underneath. Edited rows show new values, and if a selected row drops out of the filter the keep-with-notice rule applies. View choice persists long-term; the rest of the restored state is session-only.

## Forms & Editing (9)

- **REQ-032 — Full-Screen Edit Form** (must): The Edit action opens a full-screen form where all fields are editable, laid out to roughly match the read view scaled up for edit controls. A large Save button commits; leaving the page with unsaved changes warns first; Cancel reverts to the original values at any time before save.
- **REQ-033 — Field-Settings-Driven Validation** (must): Validation rules, required-ness, and defaults come from data-set field settings, never from per-form hand rules. Required fields show an asterisk on the label. Validation runs per field on exit plus a full sweep at save, which focuses the first problem; messages appear inline at the offending field.
- **REQ-034 — Smart Input Parsing and Formatting** (must): Typed fields format automatically — address, phone, email, website. Pasting free text into a composite field, such as a full address or full name, resolves it into components: confident parts fill in, the unresolved remainder stays visible, and the paste is never blocked. Entering a postal code fills city and state automatically.
- **REQ-035 — Per-Field Edit Window** (must): Double-clicking a read-only element opens a small window editing just that field, with its own Save and Cancel. Save commits that single field immediately as a single-field write with a concurrency check; Cancel discards. No full-record save is required.
- **REQ-036 — Relationship Lookup Fields** (must): Fields referencing another data set use a standard type-ahead control searching the related data set under its own permissions, with inline affordances to open the selected record and to create a new related record without leaving the form.
- **REQ-037 — Create Form and Duplicate Detection** (must): Creating a record uses the same full-screen form, empty, pre-filled from field-setting defaults, with identical validation and formatting. Key identity fields trigger a non-blocking similar-records check before save, offering side-by-side comparison and a continue-or-switch choice — never a hard block. A match on a soft-deleted record offers restoring it instead. After first save the user lands on the new record's read view; Cancel beforehand creates nothing.
- **REQ-038 — Form Keyboard Standard** (must): Control-S saves everywhere. Escape cancels a per-field window and requests close elsewhere through the dirty guard. Focus starts in the first editable field, enter never submits a multi-field form, and tab stops land only on editable fields — never labels, read-only elements, or non-input controls.
- **REQ-039 — Read-Only Field Explanation** (should): Fields that cannot be edited — computed, system-maintained, or blocked by permission — appear in their usual position rendered read-only, and clicking one explains why it is not editable.
- **REQ-040 — Field-Level Help** (should): Field settings carry optional help text rendered as a subtle affordance — an info marker on the label with text on hover or focus — never permanent hint paragraphs. The content is admin-maintained in field settings, not hardcoded.

## Workprocess Framework (2)

- **REQ-041 — Workprocess Plug-In Registration** (must): The platform lets an administrator register a custom multi-step application against one or more data sources, making it available directly in those data sources' action lists. A registration declares the display name and description, target data sources, selection contract — none, single, or multiple rows — and action classification. Permission is inherited from data-source access with no separate per-app grants.
- **REQ-042 — Workprocess Execution Frame** (must): A workprocess launches from an action list inheriting the selection, runs a defined sequence of steps that may branch on earlier answers, and commits nothing until completion — leaving cancels, guarded by the dirty warning. On completion the user gets a success confirmation, affected grids refresh, and a notification lands if it ran long. Every step carries the standard Help affordance. Internal step behavior is the workprocess author's freedom.

## Help System (1)

- **REQ-043 — Situation-Specific Help** (must): Every page shows a floating Help icon, and Help is the last item in every menu, both resolving through one admin-configured mapping from panels, data sets, and workprocesses to help URLs, with a URL pattern defaulting new pages sensibly. Help opens in a separate browser window, never navigating the working window away. An unmapped page goes to the help site's home with an explanatory note — never a dead link, never a hidden icon.

## Look & Feel (3)

- **REQ-044 — Color Template System** (must): Colors are controlled by templates with a fixed slot structure: system templates ship curated and users create their own by filling the slots — never per-element freeform styling. Layering is standard: the organization-branded default, then the user's app-wide template choice, then a view's row theme override for that grid. The launch set is Standard, Compact, Large print, and Dark.
- **REQ-045 — Conditional Formatting Rules** (should): A theme may carry conditional formatting rules: each names a condition on any data-source field using the standard operators and an effect drawn from the fixed color slots — row or cell background, text color, bold, or a small status icon. Rules evaluate in declared order and the first match wins per target. It is deliberately not a scripting language.
- **REQ-046 — Typography Scale and Contrast Guardrail** (should): One shared app-wide type scale: templates and row themes pick defined steps, never arbitrary sizes, with fonts from template font slots. User-created templates get a contrast guardrail that warns with a preview on unreadable combinations but never blocks.

## Data Platform (15)

- **REQ-047 — Time-Ordered Unique Identifiers** (must): Every record's primary identifier is a time-ordered UUID (version 7), generated at the application layer — never auto-increment integers. Creation time is recoverable from the identifier, identifiers are merge-safe, and recent records cluster at the index edge for read performance.
- **REQ-048 — Entity-Named Unique Field Names** (must): A field name identifies its entity and is unique across all tables: primary keys are entity-named, a foreign key carries the identical name as the primary key it references, and no bare generic column names exist. Structural system columns shared by every table — deletion, audit, and version columns — are the deliberate exemption, meaning the same thing everywhere.
- **REQ-049 — User-Defined Attributes** (must): Administrators add custom fields to any entity at runtime without a developer schema change. Custom values live in a document column on the entity's own table, defined by an attribute registry carrying name, type, entity, validation, options, label, and visibility. The API validates writes against the registry and serves custom attributes merged into records, so the UI is agnostic to built-in versus user-defined.
- **REQ-050 — Schema Registry** (must): Every field of every entity — built-in and user-defined — has a registry row: name, type, label, required and validation rules, option source, history-tracked flag, and visibility hints. Built-in rows are seeded from source-controlled definitions in the same migration that adds the column, with a startup check failing on drift. One metadata endpoint serves the registry and drives UI rendering, validation, duplicate-detection configuration, exports, and view columns.
- **REQ-051 — Option Lists as Data** (must): Choice-field options live in shared option set and option value tables — never database enums or check constraints. A field's registry row points at its option set and records store the option value identifier. Deactivating a value hides it from new entry while historic records still render; renaming is one row update; sets are shareable across fields.
- **REQ-052 — Soft Delete Everywhere** (must): A record is never physically deleted: every entity table carries deletion timestamp and deleting-user columns, reads exclude deleted rows by default through a central query rule, and administrators can restore. All indexes and unique constraints are partial over live rows, so live reads never pay for deleted rows and re-adding a live duplicate of a deleted row works.
- **REQ-053 — Optimistic Concurrency Control** (must): Every entity table carries an integer row version incremented on every update. Reads that can lead to edits carry the version out; writes carry it back, and a version mismatch returns a conflict response containing the current record so the client can merge or refresh. No locks — readers never wait.
- **REQ-054 — Audit Columns and Field History** (must): Every entity table carries created and modified timestamp and user columns, maintained by the API on every write. One system-wide field-change table records old and new values whenever a history-tracked field changes, with tracking a per-field flag in the schema registry rather than all fields. A record's history panel is one indexed lookup.
- **REQ-055 — Keyset List Reads and Server Search** (must): List endpoints paginate by keyset cursor — sort value plus record identifier tiebreak — never by offset, so every page is an equal-cost indexed seek stable under concurrent inserts. Counts and aggregates run as a separate parallel query so rows render first. Server-side text search uses trigram indexes on searchable columns declared per entity in the schema registry.
- **REQ-056 — Generated Read Views** (must): Application reads and admin-authored data sources target generated per-entity views, never base tables. Views join option labels in, promote custom attributes to named columns, and exclude soft-deleted rows, regenerating when the schema registry changes. Admin SQL runs under a dedicated read-only database role with a statement timeout, and user-based row filtering is injected server-side where a source declares it.
- **REQ-057 — Change Feed** (must): Clients synchronize through a change feed: a request with a watermark returns the entity type, record identifier, row version, and change kind for everything changed since, plus a new watermark, with deletes and restores appearing as change kinds. Catch-up from any older watermark is idempotent. Server push, when it arrives, upgrades only the transport — same tuples, watermark, and invalidation.
- **REQ-058 — Background Job Queue** (must): All background work runs through one job table and worker: typed jobs with a document payload, status lifecycle, attempt count, scheduled time, and a lease claimed safely by concurrent workers with crash reclaim. Endpoints that can exceed ten seconds enqueue and return a job identifier, with status queryable and completion surfacing through the change feed. Exports and prints are job types producing downloadable artifacts with retention expiry.
- **REQ-059 — Write Contract** (must): Updates send only changed fields with the row version — partial update is the primary write verb, and record creation is the only whole-record write. Every response uses one envelope of data, meta, and errors. Validation failures return structured per-field errors reporting all failures in one round trip. Duplicate detection is enforced at the API on create against normalized match columns, returning candidates for merge or an explicit recorded override.
- **REQ-060 — User Preferences Store** (must): All per-user personalization — saved grid layouts, pins, filters, navigation style, startup choice, template selection — persists in one user preference table with namespaced keys and document values, behind one get-and-put preferences API. Organization defaults are rows without a user that user rows override. No per-feature preference tables.
- **REQ-061 — Shared Normalization Services** (must): Parsing and normalization are server-side shared services: postal-code lookup backed by a refreshable reference table, address parsing, person-name parsing, and phone normalization. The same services run in API validation and feed the duplicate-detection match columns, so there is exactly one definition of equality.

## CRM Integration (3)

- **REQ-062 — CRM as System of Record for Master Data** (must): The organization's CRM remains the system of record for mentor, client, and engagement master records produced by the existing intake and staff tools. The application references those records rather than forking their truth, and changes it makes to them are written to the CRM so staff keep working in their existing tools without a second place to look.
- **REQ-063 — Application-Owned Mentoring Process Data** (must): Mentoring-process data that has no system of record today — session logs, meeting notes, next steps, progress against goals — is owned by the application's data store under the platform data standards, linked to the CRM records it concerns. Each data set's ownership side is declared explicitly at design time, and staff-relevant data remains visible to staff.
- **REQ-064 — CRM Outage Resilience** (should): When the CRM is unreachable or slow, the application degrades gracefully: reads that depend on it say so specifically instead of showing empty or stale-as-fresh data, user work in progress is preserved locally, and failed CRM writes distinguish transient faults, which retry with backoff, from terminal ones, which surface immediately with their specific cause and a retry affordance.

## Mentoring Domain (6)

- **REQ-065 — Mentor Engagement Workspace** (must): A mentor signing in sees their own mentoring engagements: each client's business profile, requested focus areas, engagement status, and the running history of sessions and next steps. This is the mentor's primary working surface for delivering mentoring.
- **REQ-066 — Assignment Acceptance** (must): When staff assign a mentor to an engagement, the mentor sees the pending assignment in the application and accepts or declines it. The decision updates the engagement's status in the system of record so staff see it in their existing tools, and a decline captures the reason.
- **REQ-067 — Session Logging** (must): A mentor records each mentoring session against an engagement: date, duration, topics covered, notes, and next steps. Sessions become the engagement's running history and the raw material for reporting on mentoring delivered.
- **REQ-068 — Engagement Progress and Outcomes** (must): Mentors and staff can update an engagement's direction over its life: revising focus areas, recording progress toward the client's goals, pausing, resuming, or ending the engagement, and capturing an outcome when it ends.
- **REQ-069 — Mentor Capacity and Availability** (should): A mentor can view and update their accepting-new-clients flag and see their capacity position — maximum concurrent clients, currently active, available. Staff assignment tooling reads the same values, so a mentor pausing intake takes effect immediately for assignment decisions.
- **REQ-070 — Leadership Reporting** (should): Leadership can see program health without asking staff to compile it: active engagements, sessions delivered over time, mentor utilization against capacity, and engagement outcomes. Views are read-only and shaped for program-level questions rather than record-level work.

## Open questions at this gate

1. **CRM boundary (the one real architecture tension).** The May L2 PRDs say the app-local store holds "plumbing only" and every business write goes synchronously through to the CRM. The July database/API standards mandate a full app-owned Postgres data layer (registry, custom attributes, change feed, jobs...). REQ-062/063 are worded to be true under either resolution: CRM master data is never forked; new mentoring-process data (sessions etc.) is app-owned and linked; the per-data-set boundary is declared at design time. The specific open sub-question: how staff see mentoring activity (write-through into new CRM entities vs a staff read surface vs exports).
2. **REQ-003 fold-in.** The June "Mentor Home Screen" candidate gets provenance edges and confirms with this set (its content is consistent with REQ-009/010/011).
3. **Release scope.** All confirmed REQs into one release, or platform-first with the Mentoring Domain (REQ-065..070) in a second release.
